Tables, Views & Storage

Guía de estudio SnowPro Core COF-C03 — Dominios 1.5, 4.2, 5.1 | Basado en documentación oficial de Snowflake

🏆 Los 10 Conceptos Más Importantes

  1. Micro-partitions — Unidades de 50-500 MB (sin comprimir), columnar, automáticas, inmutables. Son la base de todo pruning.
  2. Clustering Keys — Solo para tablas multi-TB. Máximo 3-4 columnas, de menor a mayor cardinalidad. No aplican a Hybrid Tables.
  3. Automatic Clustering — Servicio serverless. No usa tu warehouse. Se suspende/resume con ALTER TABLE. Tablas clonadas inician con AC suspendido.
  4. Tabla Permanent vs Transient vs Temporary — Diferencia clave: Time Travel (0-1 día en transient/temp, hasta 90 en permanent) y Fail-safe (0 en transient/temp, 7 en permanent).
  5. External Tables — Read-only. Datos en external stage. Columna VALUE (VARIANT) + pseudocolumnas. Se pueden particionar.
  6. Hybrid Tables — Row-store + columnar async. Primary key obligatorio. Soportan constraints enforced (UNIQUE, FK). Para workloads OLTP (Unistore).
  7. Search Optimization Service — Crea un search access path. Enterprise Edition+. Serverless. Ideal para point lookups, VARIANT queries, geospatial.
  8. Secure Views — Ocultan la definición SQL y evitan optimizaciones que podrían filtrar datos. Obligatorias para data sharing.
  9. Materialized Views — Pre-computadas, mantenidas automáticamente (serverless). Enterprise Edition+. No permiten JOINs. Soportan clustering keys.
  10. Zero-Copy Cloning — Comparte micro-partitions. Sin costo de storage adicional hasta que hay cambios (DML). Metadata-only al inicio.

Micro-partitions & Data Clustering Domain 1.5

Concepto

Snowflake divide automáticamente todos los datos de las tablas en micro-partitions: unidades contiguas de almacenamiento de entre 50 MB y 500 MB de datos sin comprimir. Internamente se almacenan comprimidos y en formato columnar. Este particionamiento es transparente — no requiere DDL ni mantenimiento manual.

Snowflake almacena metadatos por cada micro-partition: rango de valores por columna, número de valores distintos, y propiedades adicionales para optimización. Esto permite un pruning extremadamente granular.

Beneficios clave

Las micro-partitions son automáticas (no se definen manualmente), su tamaño pequeño permite DML eficiente y pruning fino, pueden tener rangos de valores que se solapan (evitando data skew), cada columna se almacena y comprime de forma independiente, y Snowflake elige el algoritmo de compresión óptimo por columna.

Impacto en operaciones

Las operaciones DML como DELETE, UPDATE y MERGE aprovechan los metadatos de micro-partitions. Algunas operaciones como borrar todas las filas son metadata-only. Cuando se hace DROP de una columna, las micro-partitions no se reescriben inmediatamente — los datos de la columna eliminada permanecen en storage.

El query pruning funciona en dos niveles: primero pruna micro-partitions completas que no coinciden con el filtro, luego escanea solo las columnas necesarias dentro de cada micro-partition seleccionada. Los predicados con subqueries no se usan para pruning.

Data Clustering & Clustering Depth

Los datos se ordenan naturalmente según cómo fueron insertados. El clustering depth mide la profundidad promedio de solapamiento de micro-partitions para un conjunto de columnas. Un depth más pequeño indica mejor clustering. Se monitorea con SYSTEM$CLUSTERING_INFORMATION y SYSTEM$CLUSTERING_DEPTH.

micro-partition 50-500 MB columnar storage pruning clustering depth metadata SYSTEM$CLUSTERING_INFORMATION SYSTEM$CLUSTERING_DEPTH
En BigQuery (GCP), el equivalente conceptual son las "partitions" y el clustering automático, pero BigQuery usa particionamiento explícito por columna (DATE, TIMESTAMP, INT). En Snowflake es totalmente automático y transparente. En AWS Redshift, el equivalente son los sort keys y distribution keys, pero requieren definición manual y VACUUM para mantenimiento.
¿Cuál es el tamaño de una micro-partition? → 50-500 MB sin comprimir.
¿Las micro-partitions requieren definición manual? → No, son automáticas.
¿Qué metadatos almacena Snowflake por micro-partition? → Rango de valores, valores distintos, propiedades de optimización.
¿Qué pasa cuando se hace DROP COLUMN? → Las micro-partitions NO se reescriben; los datos persisten en storage.

Clustering Keys & Clustered Tables Domain 4.2

Concepto

Un clustering key es un subconjunto de columnas (o expresiones) designadas para co-localizar datos en las mismas micro-partitions. Están pensados para tablas muy grandes (multi-terabyte) donde el orden natural se ha degradado por DML extensivo.

Cuándo usar clustering keys

Se debe considerar cuando: las queries son más lentas de lo esperado, el clustering depth es alto, la tabla tiene muchas micro-partitions (multi-TB), las queries son selectivas o hacen ORDER BY, y un alto porcentaje de queries se beneficiaría de las mismas columnas.

Importante: los clustering keys generan costos de compute (serverless) y storage adicional. Son más costo-efectivos en tablas consultadas frecuentemente y que cambian poco.

Estrategias de selección

Máximo 3-4 columnas. Priorizar: primero columnas usadas en filtros selectivos (ej: fecha), luego columnas de JOINs. Ordenar de menor a mayor cardinalidad. Para columnas de alta cardinalidad (ej: timestamps), usar expresiones como TO_DATE(c_timestamp) para reducir la cardinalidad. Para VARCHAR, solo se usan los primeros 5 bytes para clustering metadata.

-- Crear tabla con clustering key
CREATE TABLE t1 (c1 DATE, c2 STRING, c3 NUMBER) CLUSTER BY (c1, c2);

-- Clustering por expresiones
CREATE TABLE t2 (c1 TIMESTAMP, c2 STRING) 
  CLUSTER BY (TO_DATE(c1), SUBSTRING(c2, 0, 10));

-- Clustering por paths en VARIANT
CREATE TABLE t3 (t TIMESTAMP, v VARIANT) 
  CLUSTER BY (v:"Data":id::NUMBER);

-- Modificar clustering key
ALTER TABLE t1 CLUSTER BY (c1, c3);

-- Eliminar clustering key
ALTER TABLE t1 DROP CLUSTERING KEY;

Reclustering

El reclustering es automático (Automatic Clustering). Re-organiza micro-partitions según el clustering key. Genera nuevas micro-partitions y marca las originales como eliminadas (pero persisten para Time Travel y Fail-safe). Las micro-partitions originales se purgan después de 8-97 días.

CLUSTER BY 3-4 columnas máx menor a mayor cardinalidad TO_DATE() TRUNC() 5 bytes VARCHAR reclustering serverless credits
¿Se pueden definir clustering keys en Hybrid Tables? → NO. En Hybrid Tables los datos se ordenan por primary key.
¿Qué tipos de datos NO se pueden usar en clustering keys? → GEOGRAPHY, VARIANT, OBJECT, ARRAY (directamente; se permiten expresiones sobre paths en VARIANT).
¿Qué pasa con clustering keys al clonar una tabla? → Se copian, pero Automatic Clustering queda SUSPENDIDO en el clon.

Automatic Clustering Domain 4.2

Concepto

Servicio de Snowflake que gestiona todo el reclustering automáticamente. No requiere que designes un warehouse — usa serverless compute. Es non-blocking (no bloquea DML mientras recluster). Solo recluster si la tabla se beneficiaría.

Control

-- Suspender
ALTER TABLE t1 SUSPEND RECLUSTER;

-- Reanudar
ALTER TABLE t1 RESUME RECLUSTER;

-- Estimar costos
SELECT SYSTEM$ESTIMATE_AUTOMATIC_CLUSTERING_COSTS('db.schema.table', '(c1, c2)');

Costos

Se cobran credits por serverless compute. El storage puede aumentar por Fail-safe de micro-partitions eliminadas. Los resource monitors NO pueden controlar el crédito del warehouse AUTOMATIC_CLUSTERING. Para monitorear costos se usa AUTOMATIC_CLUSTERING_HISTORY (vista en Account Usage o función en Information Schema).

serverless SUSPEND RECLUSTER RESUME RECLUSTER non-blocking DML AUTOMATIC_CLUSTERING_HISTORY SYSTEM$ESTIMATE_AUTOMATIC_CLUSTERING_COSTS
En BigQuery, el clustering es automático y gratuito (sin costo extra de mantenimiento). En Redshift, necesitas ejecutar VACUUM manualmente o programar automatic table sort. Snowflake es intermedio: automático pero con costo serverless.
¿Automatic Clustering usa tu virtual warehouse? → NO, usa serverless compute.
¿Se puede usar Resource Monitor para controlar costos de AC? → NO.
¿Qué pasa con AC al clonar una tabla? → AC se SUSPENDE en la tabla clonada, incluso si estaba activo en la fuente.

Tipos de Tablas: Permanent, Temporary, Transient Domain 1.5

Concepto

Snowflake tiene tres tipos principales de tablas estándar que se diferencian por su persistencia, Time Travel y Fail-safe.

Temporary Tables

Solo existen en la sesión. Si se crea una temporary table con el mismo nombre que una permanente, la temporary tiene precedencia en esa sesión (oculta la permanente). No requiere privilegio CREATE TABLE en el schema. Se recomienda hacer DROP explícito antes de terminar la sesión.

Transient Tables

Sin Fail-safe = menos costos de storage. Ideal para datos transitorios que deben persistir entre sesiones. Se pueden crear schemas y databases transient, y todo dentro hereda la condición transient.

CREATE TEMPORARY TABLE tmp (id NUMBER, dt DATE);
CREATE TRANSIENT TABLE trn (id NUMBER, dt DATE);
CREATE TRANSIENT SCHEMA my_schema;   -- todo adentro será transient
CREATE TRANSIENT DATABASE my_db;     -- todo adentro será transient
TEMPORARY TRANSIENT PERMANENT Fail-safe Time Travel 0-1 vs 0-90 session-scoped
En BigQuery, las tablas temporales se crean con CREATE TEMP TABLE y expiran en 24h o al terminar el script. No hay concepto de "transient". En Redshift, las tablas temporales también son session-scoped, y no hay equivalente directo a transient con Fail-safe deshabilitado.
¿Las tablas temporales tienen Fail-safe? → NO (0 días).
¿Las tablas transient tienen Fail-safe? → NO (0 días).
¿Se puede convertir una tabla temporary a permanent? → NO.
¿Se pueden crear Hybrid Tables temporales o transient? → NO.

External Tables Domain 1.5 / 3.1

Concepto

Una external table permite consultar datos en un external stage (S3, GCS, Azure Blob) como si fueran una tabla. Snowflake almacena metadatos de archivos pero los datos permanecen fuera. Son read-only (no DML). Soportan todos los formatos de COPY INTO excepto XML.

Columnas predeterminadas

VALUE (VARIANT): contiene cada fila como variant. METADATA$FILENAME: nombre del archivo + path. METADATA$FILE_ROW_NUMBER: número de fila dentro del archivo. Se pueden definir virtual columns como expresiones sobre VALUE.

Partitioned External Tables

Se recomienda fuertemente particionar las external tables. Las particiones se pueden agregar automáticamente (expresiones sobre METADATA$FILENAME) o manualmente. La partición mejora el rendimiento al escanear solo slices relevantes.

Rendimiento

Las queries sobre external tables son más lentas que sobre tablas nativas. Para mejorar performance se pueden crear materialized views sobre external tables. Para archivos Parquet, considerar usar Apache Iceberg™ tables.

external stage read-only VALUE (VARIANT) METADATA$FILENAME METADATA$FILE_ROW_NUMBER virtual columns partitioned materialized view sobre external table
En BigQuery, el equivalente son las external tables (federated queries) sobre Cloud Storage, Drive, etc. En AWS, es como usar Athena + Glue Catalog para consultar datos en S3. La diferencia clave es que Snowflake gestiona los metadatos internamente con refresh automático via notificaciones (SQS, GCS Pub/Sub, Event Grid).
¿Se puede hacer INSERT en una external table? → NO, son read-only.
¿Qué formato NO soportan las external tables? → XML.
¿Cómo mejorar el rendimiento de queries sobre external tables? → Crear materialized views sobre ellas, o usar particionamiento.
¿Cuál es el tamaño recomendado para archivos Parquet en external tables? → 256-512 MB para archivos, 16-256 MB para row groups.

Hybrid Tables Domain 1.5

Concepto

Las Hybrid Tables son un tipo de tabla optimizado para baja latencia y alto throughput con lectura/escritura basadas en índices. Usan un row-store como storage primario con copia asíncrona a columnar (object storage). Soportan row locking para alta concurrencia y constraints enforced (UNIQUE, FK, NOT NULL).

Arquitectura

Se integran en la misma arquitectura de Snowflake: mismo query engine, mismos warehouses. Se pueden hacer JOINs con tablas estándar en la misma transacción atómica sin necesidad de two-phase commit.

Casos de uso (Unistore)

Metadata de aplicaciones con alta concurrencia de updates, serving de agregados pre-computados con baja latencia, aplicaciones transaccionales ligeras con modelo relacional.

Limitaciones clave

No soportan clustering keys (datos ordenados por PK), no se pueden crear como TEMPORARY ni TRANSIENT, no se crean dentro de schemas/databases transient, disponibles solo en AWS y Azure (no GCP al momento de la doc).

CREATE OR REPLACE HYBRID TABLE my_hybrid (
  id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  category_id INT,
  FOREIGN KEY (category_id) REFERENCES categories(id)
);
HYBRID TABLE row-store PRIMARY KEY obligatorio constraints enforced Unistore row locking OLTP + OLAP
En GCP, el equivalente sería usar Cloud Spanner o Bigtable para workloads OLTP junto con BigQuery para analytics. En AWS, sería Aurora + Redshift combinados con zero-ETL. Snowflake Hybrid Tables unifican ambos en una sola plataforma.
¿Las Hybrid Tables soportan clustering keys? → NO.
¿Las constraints (FK, UNIQUE) se enforzan en standard tables? → NO (solo informational, excepto NOT NULL). En Hybrid Tables SÍ se enforzan.
¿Qué tipo de storage primario usan? → Row-store.
¿Se pueden crear en GCP? → NO al momento (solo AWS y Azure).

Search Optimization Service Domain 4.2

Concepto

Servicio que crea y mantiene un search access path: estructura persistente que rastrea qué valores están en qué micro-partitions. Permite saltar micro-partitions en queries de tipo lookup. Requiere Enterprise Edition o superior.

Tipos de queries que beneficia

Point lookups selectivos (igualdad, IN), búsquedas de texto (SEARCH, SEARCH_IP), substring y regex (LIKE, ILIKE, RLIKE), queries sobre elementos en columnas VARIANT/OBJECT/ARRAY, queries geospatiales con GEOGRAPHY, y queries sobre tipos estructurados (structured ARRAY, OBJECT, MAP).

Cómo funciona

Un servicio de mantenimiento crea y actualiza el search access path de forma serverless. Se habilita con ALTER TABLE ... ADD SEARCH OPTIMIZATION. Se puede activar para columnas específicas o para toda la tabla.

-- Habilitar para toda la tabla
ALTER TABLE t1 ADD SEARCH OPTIMIZATION;

-- Habilitar para columnas específicas
ALTER TABLE t1 ADD SEARCH OPTIMIZATION ON EQUALITY(c1), EQUALITY(c2);

-- Deshabilitar
ALTER TABLE t1 DROP SEARCH OPTIMIZATION;

Search Optimization vs Clustering Keys

Clustering keys reorganizan datos físicamente para rangos amplios. Search Optimization crea un índice adicional para lookups puntuales. Son complementarios, no excluyentes.

search access path Enterprise Edition+ serverless point lookup EQUALITY / SUBSTRING / GEO ADD SEARCH OPTIMIZATION complementario a clustering
En BigQuery, el equivalente más cercano es el Search Index (CREATE SEARCH INDEX) para búsquedas de texto. En Redshift, no hay equivalente directo — se usan sort keys y zone maps. El search access path de Snowflake es más granular que los zone maps tradicionales.
¿Search Optimization reemplaza a clustering keys? → NO, son complementarios.
¿Qué edición requiere? → Enterprise o superior.
¿Usa tu warehouse? → NO, es serverless.
¿Funciona con datos semi-estructurados (VARIANT)? → SÍ.

Vistas Estándar (Non-Materialized) Domain 1.5

Concepto

Una vista es una definición nombrada de un query. Los resultados se generan al momento de consultar la vista — no se almacenan. Permiten segregar, combinar y proteger datos. Se pueden usar en JOINs, subqueries, etc.

Si no se especifica el schema en el CREATE VIEW, Snowflake asume que la tabla referida está en el mismo schema que la vista (no el schema activo de la sesión). Esto es importante para portabilidad.

Características

No consumen storage adicional, no tienen costo de mantenimiento, no mejoran performance per se (se re-ejecuta el query cada vez), pero simplifican la lógica de queries y proporcionan seguridad a nivel de fila/columna.

CREATE VIEW no almacena datos re-ejecuta query segregación de datos schema de la vista

Secure Views Domain 1.5 / 2.2

Concepto

Las Secure Views desactivan optimizaciones internas que podrían exponer datos ocultos y esconden la definición SQL de la vista. Dos propósitos: privacidad de datos y protección de lógica de negocio.

Cuándo usar

Para vistas que controlan acceso a datos sensibles. Obligatorias para Secure Data Sharing. NO se deben usar solo por conveniencia de queries — pueden ser más lentas por las optimizaciones deshabilitadas.

Cómo se expone data en vistas no-seguras

A través de user code que explota optimizaciones internas. Ejemplo: un query con una division-by-zero condicional podría revelar la existencia de datos filtrados.

-- Crear secure view
CREATE SECURE VIEW my_secure_view AS
  SELECT col1, col2 FROM sensitive_table WHERE region = 'US';

-- Convertir vista existente a secure
ALTER VIEW my_view SET SECURE;

-- Quitar secure
ALTER VIEW my_view UNSET SECURE;
SECURE VIEW definición oculta optimizaciones deshabilitadas data sharing obligatorio SET SECURE / UNSET SECURE
¿Cuándo es obligatorio usar Secure Views? → Al compartir datos via Secure Data Sharing.
¿Quién puede ver la definición de una Secure View? → Solo el rol owner de la vista.
¿Las Secure Views son más rápidas que las normales? → NO, pueden ser más lentas.

Materialized Views Domain 4.2

Concepto

Una Materialized View (MV) almacena resultados pre-computados de un query. Más rápida que re-ejecutar el query. Mantenimiento automático y transparente por un servicio background de Snowflake. Los datos siempre están actualizados — si hay cambios, Snowflake combina MV cache con datos frescos del base table. Requiere Enterprise Edition.

Cuándo crear

Cuando los resultados cambian poco, se usan frecuentemente, el query es costoso, o la vista es sobre una external table. Las MVs soportan clustering keys.

Limitaciones importantes

Solo pueden definirse sobre una sola tabla (no JOINs). El mantenimiento automático consume credits (serverless). Usan storage adicional. No se soportan todas las funciones SQL.

En BigQuery, las materialized views también existen con auto-refresh y smart tuning automático. En Redshift, las materialized views requieren REFRESH MATERIALIZED VIEW manual o auto-refresh configurado. Snowflake las mantiene automáticamente sin intervención.
¿Las Materialized Views permiten JOINs? → NO, solo una tabla base.
¿Requieren warehouse para mantenimiento? → NO, usan serverless compute.
¿Qué edición necesitan? → Enterprise o superior.
¿Cuál es la diferencia clave entre MV y Dynamic Table? → MV: single table, serverless, no JOINs. DT: multi-table, warehouse, soporta JOINs.

Views vs Materialized Views vs Dynamic Tables Domain 1.5 / 4.2

Concepto

Snowflake ofrece tres mecanismos principales para transformar/presentar datos. La elección depende de si necesitas performance (MV, DT), seguridad (views, secure views), JOINs (views, DT), o pipelines ELT (DT).

Views: definición de query, no almacenan nada, sin costo de storage/maintenance. Materialized Views: resultados pre-computados, single table, serverless maintenance, Enterprise+. Dynamic Tables: transformaciones declarativas sobre múltiples tablas, mantenidas por un warehouse que tú especificas, target lag configurable.

Table Design Considerations Domain 1.5

Concepto

Mejores prácticas para diseñar tablas en Snowflake, enfocadas en eficiencia de storage y query performance.

Tipos de datos

Usar DATE/TIMESTAMP en lugar de VARCHAR para fechas — más eficiente en storage y performance. Elegir la granularidad correcta (DATE vs TIMESTAMP_NTZ/LTZ/TZ).

Constraints

En tablas estándar, las constraints (PK, FK, UNIQUE) son informational only — NO se enforzan excepto NOT NULL. En Hybrid Tables SÍ se enforzan. Las constraints informacionales son útiles para el query optimizer y documentación.

Otros tips

Elegir el tipo de tabla correcto (permanent, transient, temporary) según la necesidad de data protection. Considerar el impacto de Time Travel y Fail-safe en costos de storage. Usar NUMBER con precisión y escala adecuada en lugar de tipos más amplios.

DATE/TIMESTAMP vs VARCHAR constraints informational NOT NULL enforced Hybrid = enforced constraints
¿Las PK/FK se enforzan en tablas estándar? → NO, son informational. Solo NOT NULL se enforza.
¿Por qué usar DATE en lugar de VARCHAR? → Mejor compression, mejor performance de queries, menos storage.

Cloning Considerations Domain 5.1

Concepto

El zero-copy cloning de Snowflake crea una copia de metadatos que referencia las mismas micro-partitions subyacentes. No se duplican datos al momento del clon — solo se incurre en storage adicional cuando los datos divergen (DML en clon o fuente).

Privilegios en clones

Al clonar un database o schema, los privilegios de los objetos hijos se heredan. El clon del container (DB/schema) en sí NO hereda sus privilegios. Se puede usar COPY GRANTS en CREATE TABLE/VIEW CLONE para copiar privilegios.

Consideraciones clave

Clustering keys se copian pero Automatic Clustering queda SUSPENDIDO. Las secuencias default se referencian al clon si están en el mismo schema/db clonado, sino apuntan a la fuente. Los pipes clonados: el rol creador obtiene OWNERSHIP. Los clones de transient tables desde permanent tables comparten micro-partitions y pueden retrasar Fail-safe de la fuente.

-- Clonar tabla
CREATE TABLE t1_clone CLONE t1;

-- Clonar con Time Travel (punto en el tiempo)
CREATE TABLE t1_clone CLONE t1 AT(TIMESTAMP => '2024-01-01 00:00:00');

-- Clonar database completo
CREATE DATABASE db_clone CLONE my_db;
zero-copy clone metadata-only CLONE COPY GRANTS AC suspendido en clon Time Travel clone (AT / BEFORE)
En BigQuery, el equivalente es CREATE TABLE COPY o table snapshots, pero siempre generan una copia lógica que eventualmente consume storage. En Redshift, no hay zero-copy clone nativo — necesitas UNLOAD/COPY o snapshots de cluster. La ventaja de Snowflake es que el clon es instantáneo y sin costo hasta divergencia.
¿Un clone consume storage adicional inmediatamente? → NO, solo cuando los datos cambian.
¿Los privilegios del database/schema se heredan al clonar? → NO para el container, SÍ para los objetos hijos.
¿Se puede clonar usando Time Travel? → SÍ, con AT o BEFORE.
¿Qué pasa con las clustering keys al clonar? → Se copian pero AC queda suspendido.

Data Storage & Continuous Data Protection Domain 5.1

Concepto

El storage en Snowflake pasa por tres estados secuenciales del ciclo de vida CDP: Active → Time Travel → Fail-safe. Se cobra por los tres estados. Los datos se purgan solo después de completar Fail-safe (mínimo 8 días para standard, hasta 97 días con extended Time Travel en Enterprise+).

Time Travel

Permite acceder a datos históricos. Permanent tables: hasta 90 días (Enterprise+). Transient/Temporary: máximo 1 día. Se controla con DATA_RETENTION_TIME_IN_DAYS.

Fail-safe

7 días adicionales después de Time Travel para tablas permanentes. Solo accesible por Snowflake Support (no por el usuario). Transient y temporary tienen 0 días de Fail-safe.

Monitoreo de storage

ACCOUNTADMIN puede ver storage total en Snowsight (Admin → Cost Management). Para tablas individuales: TABLE_STORAGE_METRICS (Information Schema) con columnas ACTIVE_BYTES, TIME_TRAVEL_BYTES, FAILSAFE_BYTES.

Staged files

Archivos en internal stages no tienen Time Travel ni Fail-safe, pero sí costos de storage estándar. Se recomienda eliminarlos después de cargar datos con REMOVE o la opción PURGE en COPY INTO.

Zero-copy cloning y storage

Las tablas clonadas comparten micro-partitions con la fuente. Storage adicional se incurre solo con cambios (DML). TABLE_STORAGE_METRICS distingue entre owned storage y referenced storage.

Active → Time Travel → Fail-safe DATA_RETENTION_TIME_IN_DAYS TABLE_STORAGE_METRICS ACTIVE_BYTES TIME_TRAVEL_BYTES FAILSAFE_BYTES PURGE / REMOVE
¿Cuántos días de Fail-safe tienen las tablas permanentes? → 7 días.
¿Quién puede acceder a datos en Fail-safe? → Solo Snowflake Support.
¿Los staged files tienen Fail-safe? → NO.
¿Cómo se monitorean costos de storage por tabla? → TABLE_STORAGE_METRICS view.

🪤 Trampas y Confusiones Frecuentes del Examen

Clustering keys en Hybrid Tables — Las Hybrid Tables NO soportan clustering keys. Los datos se ordenan por primary key. Si te preguntan sobre clustering y Hybrid Tables juntos, la respuesta es que no es compatible.
Constraints enforced vs informational — En tablas estándar, PK/FK/UNIQUE son INFORMATIONAL (no se enforzan). En Hybrid Tables SÍ se enforzan. NOT NULL se enforza en ambos tipos. Pregunta típica: "¿Qué tipo de constraint se enforza en standard tables?" → Solo NOT NULL.
Fail-safe en Transient vs Temporary — Ambas tienen 0 días de Fail-safe. La diferencia es la persistencia: temporary = solo sesión, transient = hasta DROP explícito.
Automatic Clustering al clonar — Siempre queda SUSPENDIDO en el clon, sin importar el estado en la fuente. Cambiar el clustering key también resume AC automáticamente (y puede generar costos).
Materialized Views vs Dynamic Tables — MV: single table, no JOINs, serverless, Enterprise+. DT: multi-table, JOINs, usa warehouse, Standard edition. No confundir cuál soporta JOINs.
Search Optimization vs Clustering — NO son lo mismo. Clustering organiza datos físicamente. Search Optimization crea un índice adicional (search access path). Son complementarios. Ambos usan serverless compute.
External Tables y DML — Son read-only. No se puede hacer INSERT, UPDATE, DELETE. Si necesitas escribir, debes hacer COPY INTO una tabla interna.
Secure Views y performance — Las Secure Views pueden ser MÁS LENTAS porque desactivan optimizaciones del query optimizer. No usarlas solo por conveniencia.
DROP COLUMN y storage — Hacer DROP COLUMN NO reescribe las micro-partitions. Los datos de la columna eliminada permanecen en storage hasta que se purguen naturalmente.
Resource Monitors y servicios serverless — Los Resource Monitors NO controlan el consumo de Automatic Clustering, Search Optimization ni Materialized View maintenance. Solo controlan warehouses de usuario.
Temporary table con mismo nombre — Si creas una temporary table con el mismo nombre que una permanent en el mismo schema, la temporary toma precedencia en la sesión. La permanent queda "oculta". Cuidado con DDL como DROP TABLE en este escenario.
VARCHAR en clustering keys — Solo se usan los primeros 5 bytes. Si los primeros caracteres son iguales para todas las filas, usar SUBSTRING para empezar después de los caracteres comunes.
