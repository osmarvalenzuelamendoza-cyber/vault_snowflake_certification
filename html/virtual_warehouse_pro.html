Virtual Warehouses

Dominio 1.4 · Peso: parte del 31% del examen · Fuente: documentación oficial Snowflake

Top 10 Conceptos Clave para el Examen

  1. Scale Up (resize) mejora performance de queries complejos; Scale Out (multi-cluster) mejora concurrencia.
  2. Los créditos se duplican con cada tamaño superior — billing es per-second con mínimo de 60 segundos.
  3. Multi-cluster Warehouses son feature de Enterprise Edition o superior.
  4. Dos modos multi-cluster: Maximized (min = max) y Auto-scale (min < max).
  5. Dos Scaling Policies: Standard (prioriza performance, inicia clusters rápido) y Economy (prioriza costo, espera 6 min de carga).
  6. Snowpark-Optimized = más memoria por nodo (16x default), ideal para ML training con Snowpark.
  7. Gen2 Standard = hardware más nuevo, mejor performance para analytics; se configura con RESOURCE_CONSTRAINT = STANDARD_GEN_2.
  8. Query Acceleration Service (QAS) = recurso serverless que acelera outlier queries con scans grandes; Enterprise Edition, scale factor default = 8.
  9. El warehouse cache se pierde al suspender — tradeoff entre ahorro de créditos vs warm cache.
  10. Aumentar el tamaño del warehouse NO mejora necesariamente la carga de datos — depende más del número y tamaño de archivos.

1. Tipos de Warehouses

Snowflake ofrece distintos tipos de virtual warehouses, cada uno optimizado para diferentes cargas de trabajo. El examen evalúa que conozcas cuándo usar cada tipo.

Standard Warehouses: Gen1 vs Gen2

Existen dos generaciones de warehouses estándar. Gen1 es el original; Gen2 es la versión de próxima generación con hardware más rápido y optimizaciones de software para operaciones como DELETE, UPDATE, MERGE y scans de tablas.

Concepto técnico

La generación se controla con la propiedad RESOURCE_CONSTRAINT que acepta los valores STANDARD_GEN_1 o STANDARD_GEN_2. Este parámetro solo se configura vía SQL, no está disponible en Snowsight. Gen2 no está disponible para tamaños 5XL y 6XL.

Palabras clave

RESOURCE_CONSTRAINT · STANDARD_GEN_1 · STANDARD_GEN_2 · WAREHOUSE_TYPE = 'STANDARD'

Analogía GCP / AWS

Es comparable a elegir entre generaciones de instancias en AWS (ej: m5 → m6i → m7i) o en GCP (N2 → N2D → C3). Hardware más nuevo, mejor relación precio/performance, misma API. En Snowflake simplemente se cambia el RESOURCE_CONSTRAINT sin gestionar infraestructura.

Posible pregunta de examen

¿Cómo se convierte un warehouse Gen1 a Gen2? → ALTER WAREHOUSE SET RESOURCE_CONSTRAINT = STANDARD_GEN_2. ¿Se puede hacer con el warehouse corriendo? → Sí, pero las queries existentes terminan en Gen1 y las nuevas usan Gen2 (se cobra por ambos recursos mientras tanto).

Snowpark-Optimized Warehouses

Proporcionan más memoria por nodo que un warehouse estándar. El default es 16x más memoria (MEMORY_16X). Están diseñados para cargas de trabajo que usan Snowpark: ML training con stored procedures en Python/Java, UDFs y UDTFs con alto consumo de memoria.

Concepto técnico

Se crean con WAREHOUSE_TYPE = 'SNOWPARK-OPTIMIZED'. El parámetro RESOURCE_CONSTRAINT controla la cantidad de memoria: MEMORY_1X (16GB), MEMORY_16X (256GB, default), MEMORY_64X (1TB, preview solo AWS). Variantes con sufijo _x86 fuerzan arquitectura x86. Su creación y reanudación pueden tardar más que un warehouse estándar.

Analogía GCP / AWS

Similar a elegir instancias memory-optimized en AWS (familia r6i/r7i) o GCP (familia M3) versus instancias de propósito general. El concepto es el mismo: más RAM por nodo para workloads que lo requieren, a mayor costo por crédito.

Posible pregunta de examen

Un equipo necesita ejecutar un modelo de ML training con Snowpark Python que consume mucha memoria. ¿Qué tipo de warehouse deben usar? → Snowpark-Optimized. ¿Los workloads que NO usan Snowpark se benefician? → No necesariamente.

Interactive Warehouses (Referencia)

Son un tipo nuevo, GA solo en algunas regiones AWS. Optimizados para queries de baja latencia y alta concurrencia con Interactive Tables. Dado que el examen no suele evaluar features que no son GA global, conoce su existencia pero no es prioritario para el estudio.

Modelo de Billing Per-Second

Concepto técnico

Snowflake cobra por segundo con un mínimo de 60 segundos cada vez que el warehouse se inicia/reanuda. Después de esos 60 segundos, cada segundo adicional se cobra de forma proporcional. Si un warehouse corre 61 segundos, se cobran exactamente 61 segundos. Si corre 30 segundos, se cobran 60. Si se suspende y reinicia, el mínimo de 60 segundos aplica nuevamente.

Posible pregunta de examen

Un warehouse corre por 61 segundos, se suspende, y luego corre otros 45 segundos. ¿Cuántos segundos se cobran? → 60 + 1 + 60 = 121 segundos (el segundo inicio tiene su propio mínimo de 60s).

Tip para el examen

Recuerda: aumentar el tamaño del warehouse NO mejora necesariamente la carga de datos. La performance de data loading depende más del número y tamaño de archivos. Un warehouse Small o Medium suele ser suficiente a menos que cargues cientos o miles de archivos en paralelo.

3. Scale Up vs Scale Out

Scale Up (Resize / Sizing Up-Down)

Resize significa cambiar el tamaño del warehouse (ej: Medium → Large). Esto le da más recursos de cómputo a cada query individual. Es la estrategia para mejorar la performance de queries complejos y lentos.

Concepto técnico

Se puede hacer con el warehouse corriendo. Los recursos nuevos NO impactan queries en ejecución; solo benefician a queries en cola y nuevos. Al reducir el tamaño (scale down), los recursos removidos pierden su cache, lo que puede afectar temporalmente la performance (igual que suspender).

Scale Out (Multi-cluster / Scaling In-Out)

Scale out significa agregar más clusters al warehouse (multi-cluster). Cada cluster tiene el mismo tamaño. Esto mejora la concurrencia — permite atender más usuarios/queries simultáneos sin que se encolen.

Regla de oro para el examen

Scale Up → para queries lentos/complejos (mejora performance individual).
Scale Out → para muchos usuarios concurrentes (mejora concurrencia).
Multi-cluster NO mejora queries lentos individuales ni la carga de datos.

Analogía GCP / AWS

Scale Up = cambiar el tipo de instancia de EC2/GCE a uno más grande (vertical scaling).
Scale Out = agregar más instancias detrás de un Load Balancer / Auto Scaling Group (horizontal scaling). En Snowflake, el multi-cluster warehouse es el "auto scaling group" nativo.

4. Multi-cluster Warehouses (Enterprise Edition)

Modos: Maximized vs Auto-scale

Concepto técnico

Maximized: MIN_CLUSTER_COUNT = MAX_CLUSTER_COUNT (ambos > 1). Todos los clusters arrancan al iniciar el warehouse. Útil cuando la carga es predecible y alta constantemente.

Auto-scale: MIN_CLUSTER_COUNT < MAX_CLUSTER_COUNT. Snowflake agrega y remueve clusters dinámicamente según la carga. Snowflake recomienda empezar con max=2 o 3, min=1 e ir ajustando.

El límite máximo de clusters depende del tamaño del warehouse. Para tamaños XS a Medium es 300, pero para 4XL-6XL es solo 10. En Snowsight el máximo configurable es 10; para más de 10 se usa SQL.

Posible pregunta de examen

¿En qué modo un multi-cluster warehouse inicia todos los clusters al arrancar? → Maximized (min = max). ¿Cuál escala automáticamente según demanda? → Auto-scale (min < max). ¿Qué edición se requiere? → Enterprise Edition o superior.

Scaling Policies (Standard vs Economy)

Solo aplican en modo Auto-scale. Controlan cuándo Snowflake agrega o remueve clusters.

PolicyCuándo inicia un cluster nuevoCuándo apaga un cluster idlePrioriza
Standard (default) Cuando una query se encola o Snowflake estima que no hay recursos suficientes para queries adicionales Después de un período sostenido de baja carga, cuando las queries en el cluster terminan Performance / Throughput
Economy Solo si estima que hay suficiente carga para mantener el cluster ocupado al menos 6 minutos Cuando estima que al cluster le quedan menos de 6 minutos de trabajo Costo / Ahorro de créditos
Palabra clave para el examen

El número mágico de Economy es 6 minutos. Si te preguntan por una policy que prioriza ahorro de créditos y puede generar que queries se encolen más → Economy. Si preguntan por minimizar queuing → Standard.

Cálculo de Créditos en Multi-cluster

Los créditos se calculan multiplicando: créditos del tamaño × número de clusters corriendo × tiempo. Si un warehouse Medium (4 cr/h) corre 3 clusters por 1 hora = 12 créditos. Si en modo Auto-scale solo 1 cluster corre la primera hora y 3 la segunda, el costo total varía por hora.

Posible pregunta de examen

Un 3X-Large multi-cluster warehouse corre 1 cluster por 1 hora, luego 2 clusters por 1 hora. ¿Total de créditos? → 64 + 128 = 192 créditos. El examen puede presentar escenarios de cálculo similares.

5. Auto-Suspend y Auto-Resume

Concepto técnico

Auto-Suspend: Suspende el warehouse tras un período de inactividad configurable. Snowflake recomienda valores bajos (5-10 min o menos) para ahorrar créditos, pero debe coincidir con los patrones de uso. Si hay gaps de 2-3 min entre queries, no pongas auto-suspend en 1 min porque el warehouse estará en ciclo constante de suspend/resume (cada resume = 60s mínimo de billing).

Auto-Resume: Reanuda automáticamente el warehouse cuando llega un query. Habilitado por defecto. Al desactivarlo, se controla manualmente cuándo corre el warehouse (control de costos).

Para multi-cluster: Auto-suspend aplica solo cuando el número mínimo de clusters está corriendo y no hay actividad. Auto-resume aplica solo cuando el warehouse completo está suspendido (ningún cluster corriendo).

Tip de configuración

Para desactivar auto-suspend vía SQL se usa AUTO_SUSPEND = 0 o NULL. Pero cuidado: un warehouse corriendo 24/7 en XL genera 16 × 24 = 384 créditos/día.

Posible pregunta de examen

¿Cuál es el riesgo de poner auto-suspend en 1 minuto cuando hay gaps de 2-3 min entre queries? → Ciclo constante de suspend/resume, cada resume tiene mínimo 60s de billing, lo que puede ser más costoso que dejar el warehouse corriendo.

6. Query Acceleration Service (QAS)

Concepto técnico

QAS es un servicio serverless que acelera partes de queries que consumen muchos recursos ("outlier queries") offloading trabajo a recursos compartidos de cómputo. Requiere Enterprise Edition. Se habilita por warehouse con ENABLE_QUERY_ACCELERATION = TRUE.

Beneficia especialmente a: analytics ad-hoc, queries con scans grandes y filtros selectivos, workloads con volumen impredecible por query. Los comandos que puede acelerar son: SELECT, INSERT, CTAS, COPY INTO <table>.

Scale Factor

El scale factor es un mecanismo de control de costos que limita cuántos recursos adicionales puede usar QAS. Funciona como multiplicador del tamaño del warehouse. Default = 8. Si se pone en 0, no hay límite. Ejemplo: un warehouse Medium (4 cr/h) con scale factor 5 puede consumir hasta 20 créditos adicionales por hora en QAS.

Palabras clave

ENABLE_QUERY_ACCELERATION · QUERY_ACCELERATION_MAX_SCALE_FACTOR · SYSTEM$ESTIMATE_QUERY_ACCELERATION() · QUERY_ACCELERATION_ELIGIBLE (vista en ACCOUNT_USAGE)

Razones de inelegibilidad para QAS

No todas las queries son elegibles. Razones comunes: no hay suficientes particiones para escanear, los filtros no son lo bastante selectivos, la cardinalidad del GROUP BY es demasiado alta, o la query usa funciones no determinísticas como SEQ o RANDOM.

Analogía GCP / AWS

No hay un equivalente directo en GCP o AWS. Lo más cercano sería el concepto de "burst capacity" o "serverless compute pooling" — recursos compartidos que se asignan temporalmente para acelerar queries pesados sin tener que redimensionar tu instancia permanente. En Redshift, lo más cercano es "Concurrency Scaling", pero QAS trabaja a nivel de query individual, no de concurrencia.

Posible pregunta de examen

¿Qué servicio de Snowflake acelera outlier queries offloading a recursos serverless? → Query Acceleration Service. ¿Cuál es el scale factor default? → 8. ¿Qué edición requiere? → Enterprise. ¿QAS puede trabajar junto con Search Optimization Service? → Sí, primero SOS pruna particiones, luego QAS acelera el scan restante.

7. Warehouse Cache

Concepto técnico

Cada warehouse mantiene un cache de datos de tabla mientras está corriendo. A mayor tamaño del warehouse, mayor cache. El cache mejora performance de queries subsecuentes que acceden a los mismos datos. El cache se pierde completamente al suspender el warehouse. Al reanudar, las primeras queries pueden ser más lentas hasta que el cache se reconstruya.

Tradeoff clave

Suspender = ahorra créditos pero pierde cache. Dejar corriendo = mantiene cache caliente pero consume créditos. También al hacer scale down (reducir tamaño) se pierde el cache de los recursos removidos.

Posible pregunta de examen

¿Qué sucede con el cache del warehouse cuando se suspende? → Se pierde. ¿Y cuando se reduce el tamaño? → El cache de los recursos removidos también se pierde. Este es el "warehouse cache" — no confundir con "query result cache" (que vive en la capa de Cloud Services) ni con "metadata cache".

8. Monitoreo de Carga del Warehouse

Concepto técnico

Snowsight ofrece un gráfico de Warehouse Activity que muestra el query load (carga promedio de queries) en intervalos configurables de hasta 2 semanas. El query load se calcula como: suma del tiempo de ejecución de todas las queries en el intervalo ÷ duración total del intervalo.

Los estados visibles en el gráfico son: Running, Queued (por sobrecarga), Queued (Provisioning) (esperando recursos al reanudar), y Blocked (por locks de transacciones).

Para ver el gráfico se necesita el privilegio MONITOR en el warehouse. También se puede usar la vista SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY para análisis más detallados de performance.

Posible pregunta de examen

Un administrador nota que queries están lentos. El gráfico de carga muestra alto queuing. ¿Qué debe hacer? → Si hay queuing por sobrecarga: agregar multi-cluster o crear un warehouse separado. Si la carga running es baja pero el query es lento: resize (scale up) el warehouse.

9. Best Practices por Caso de Uso

Ad-hoc Queries

Usar warehouses de tamaño apropiado a la complejidad de los queries. Agrupar workloads similares en el mismo warehouse. Para queries impredecibles, considerar habilitar QAS. Configurar auto-suspend bajo (5-10 min).

Data Loading

Un warehouse Small-Large suele ser suficiente a menos que se carguen cientos de archivos en paralelo. Usar un warehouse dedicado para carga (separado de queries) es una best practice. El tamaño del warehouse tiene menos impacto en performance de carga que el número y tamaño de archivos.

BI y Reporting

Cargas predecibles y concurrentes → candidato ideal para multi-cluster en modo Auto-scale con policy Standard. Si el patrón es estable, considerar Maximized. Separar warehouses por equipo o por tipo de workload para poder monitorear y optimizar independientemente.

Diferentes Equipos / Alta Concurrencia / Queries Complejos

Best practices según el temario

Diferentes equipos: usar warehouses separados por equipo/dominio para aislar costos y performance.
Alta concurrencia: multi-cluster Auto-scale o crear warehouses adicionales.
Queries complejos: scale up (warehouse más grande), habilitar QAS si los queries tienen scans grandes.

Trampas y Confusiones Frecuentes del Examen

Trampa #1 — Resize vs Multi-cluster

Resize (scale up) mejora performance de queries individuales. Multi-cluster (scale out) mejora concurrencia. Nunca los mezcles: multi-cluster NO acelera un query lento individual, y resize NO resuelve problemas de demasiados usuarios concurrentes.

Trampa #2 — Warehouse más grande = carga de datos más rápida

FALSO. Aumentar el tamaño del warehouse no garantiza mejor performance en data loading. Depende del número y tamaño de los archivos. Usar un XL para cargar 10 archivos CSV es desperdiciar créditos.

Trampa #3 — Multi-cluster requiere Enterprise

Sí, multi-cluster warehouses son feature de Enterprise Edition o superior. Lo mismo aplica para Query Acceleration Service. El examen puede poner opciones que involucren estas features en contextos de Standard Edition.

Trampa #4 — Resize mientras corre el warehouse

Es posible hacer resize con el warehouse corriendo. Pero los recursos nuevos NO benefician queries ya en ejecución — solo queries en cola y nuevos. El examen puede preguntar si un query actualmente lento se acelera al hacer resize → NO, solo nuevos queries.

Trampa #5 — Confundir los 3 tipos de cache

Query Result Cache: almacena resultados exactos de queries (capa Cloud Services, 24h, no requiere warehouse). Metadata Cache: info de objetos (row count, min/max), gestionado por Cloud Services. Warehouse Cache (local disk cache): datos de tablas almacenados en el SSD del compute, se pierde al suspender. El examen los diferencia.

Trampa #6 — Scaling Policy: el número 6

La policy Economy usa el umbral de 6 minutos para decidir si agrega o remueve clusters. Standard no tiene ese umbral — agrega clusters inmediatamente al detectar queuing. Si te preguntan por una policy que espera antes de agregar clusters → Economy.

Trampa #7 — QAS Scale Factor default

El scale factor default de QAS es 8, no 10 ni ilimitado. Un scale factor de 0 significa sin límite. El costo de QAS se cobra por separado del warehouse, como recurso serverless, por segundo.

Trampa #8 — Auto-suspend en multi-cluster

Auto-suspend aplica al warehouse completo, no a clusters individuales. El warehouse se suspende solo cuando el número mínimo de clusters está corriendo y no hay actividad. Auto-resume aplica solo cuando ningún cluster está corriendo.

Trampa #9 — Snowpark-Optimized vs Standard para SQL normal

Si un workload es puro SQL/queries analíticos y NO usa Snowpark (Python/Java UDFs, stored procedures, ML), un Snowpark-Optimized warehouse no aporta beneficio y consume más créditos por hora. El examen puede presentar escenarios donde la respuesta correcta es Standard.

Trampa #10 — Billing al hacer resize de 5XL/6XL a 4XL o menor

Al reducir de 5XL/6XL a 4XL o inferior, hay un breve período de cobro doble mientras los recursos del warehouse anterior se apagan (quiescing). Esto NO ocurre en otros downsizes.
