Virtual Warehouses

Dominio 1.4 (31%) + Dominio 4.0 (21%) — Guía consolidada desde la documentación oficial de Snowflake

🏆 Top 10 Conceptos que DEBES Dominar

  1. Créditos se duplican con cada tamaño superior de warehouse (XS=1, S=2, M=4 … 6XL=512 créditos/hora).
  2. Scale Up (resize) mejora rendimiento de queries complejas; Scale Out (multi-cluster) mejora concurrencia.
  3. Multi-cluster requiere Enterprise Edition. Dos modos: Maximized (min=max) y Auto-scale (min < max).
  4. Scaling Policies: Standard (favorece arrancar clusters rápido) vs Economy (espera 6 min de carga para arrancar).
  5. Billing: per-second con mínimo de 60 segundos cada vez que se resume el warehouse.
  6. Warehouse cache se pierde al suspender. Trade-off: ahorro de créditos vs pérdida de cache.
  7. Query Acceleration Service (QAS) es Enterprise Edition, usa recursos serverless, el scale factor controla costos.
  8. Snowpark-optimized ofrece hasta 16x memoria por nodo (default). Ideal para ML training con Snowpark.
  9. Gen2 warehouses usan hardware más rápido. Se configuran con RESOURCE_CONSTRAINT = STANDARD_GEN_2.
  10. Aumentar tamaño NO mejora data loading significativamente; depende más del número/tamaño de archivos.

Tipos de Warehouse

Snowflake tiene tres tipos principales de warehouse según su propósito:

Standard (Gen1 y Gen2)

El tipo por defecto. Gen1 es la primera generación. Gen2 es la nueva generación con hardware más rápido y optimizaciones de software para analytics y data engineering. Se especifica con RESOURCE_CONSTRAINT = STANDARD_GEN_2. Gen2 no está disponible para tamaños 5XL ni 6XL.

Snowpark-Optimized

Warehouse con memoria extendida (hasta 16x por defecto, configurable a 64x/1TB). Diseñado para cargas de trabajo Snowpark con alto consumo de memoria como ML training. Se crea con WAREHOUSE_TYPE = 'SNOWPARK-OPTIMIZED'. Consume más créditos que un standard del mismo tamaño.

Interactive (GA en regiones AWS selectas)

Optimizado para baja latencia y alta concurrencia. Trabaja con interactive tables. Timeout de query por defecto: 5 segundos. No soporta auto-suspend. No puede consultar tablas estándar. Actualmente no es testeable en el examen según la guía, pero conocerlo es útil.

AWS: Standard ≈ Redshift Serverless general, Snowpark-Optimized ≈ Redshift con nodos ra3 high-memory, Interactive ≈ DynamoDB/ElastiCache para baja latencia.
GCP: Standard ≈ BigQuery slots, Snowpark-Optimized ≈ Dataproc con high-memory machines para Spark ML.

¿Cuál es la diferencia entre un warehouse Standard y un Snowpark-Optimized? → El Snowpark-Optimized provee más memoria por nodo (16x por defecto), ideal para operaciones de ML y UDFs con Snowpark. Consume más créditos por hora.

Aumentar el tamaño del warehouse NO mejora necesariamente el data loading. La performance de carga depende más del número y tamaño de archivos. Para cargas bulk, un warehouse Small/Medium suele ser suficiente a menos que cargues cientos de archivos concurrentemente.

Gen1 vs Gen2 Standard Warehouses

¿Qué es Gen2?

La siguiente generación de warehouses standard con hardware más rápido y optimizaciones de software para operaciones de delete, update, merge y table scan. Se configura con:

CREATE WAREHOUSE my_wh RESOURCE_CONSTRAINT = STANDARD_GEN_2;
-- Convertir Gen1 a Gen2:
ALTER WAREHOUSE my_wh SET RESOURCE_CONSTRAINT = STANDARD_GEN_2;

Consideraciones clave

  • Gen2 NO disponible para tamaños 5XL y 6XL
  • No aplica a Snowpark-optimized warehouses
  • Al convertir un warehouse activo de Gen1→Gen2, las queries existentes terminan en Gen1, las nuevas van a Gen2. Se cobra por ambos durante la transición
  • Si usas replication, las regiones secundarias deben soportar Gen2
  • El default depende de tu región y fecha de creación de la organización

¿Cómo cambias un warehouse existente de Gen1 a Gen2? → ALTER WAREHOUSE SET RESOURCE_CONSTRAINT = STANDARD_GEN_2. Se puede hacer con el warehouse corriendo o suspendido.

Snowpark-Optimized Warehouses

Cuándo usarlo

  • ML training con stored procedures en Snowpark (Python, Scala, Java)
  • UDFs/UDTFs con alto consumo de memoria
  • Cargas que NO son Snowpark probablemente no se benefician

Scaling Policies (Multi-Cluster)

Solo aplican en modo Auto-scale (min ≠ max clusters).

PolicyArranca nuevo cluster cuando…Apaga cluster inactivo cuando…Prioriza
Standard (default)Se detecta queuing O se estima que no hay recursos suficientes para queries nuevasDespués de un periodo sostenido de baja cargaPerformance / evitar queuing
EconomySe estima que hay carga suficiente para mantener el cluster ocupado al menos 6 minutosSe estima que el cluster tiene menos de 6 minutos de trabajo restanteAhorro de créditos

¿Cuál scaling policy favorece el ahorro de créditos sobre la velocidad de respuesta? → Economy. Solo arranca un cluster si estima al menos 6 minutos de carga, lo que puede causar más queuing.

La policy Legacy ya fue eliminada. Todos los warehouses que la usaban ahora usan Standard automáticamente. Si ves Legacy como opción en el examen, descártala.

Multi-Cluster Warehouses

Requiere Enterprise Edition

Se define con MAX_CLUSTER_COUNT > 1. Los créditos se multiplican por el número de clusters activos.

CREATE WAREHOUSE my_mcw
  WAREHOUSE_SIZE = 'MEDIUM'
  MAX_CLUSTER_COUNT = 5
  MIN_CLUSTER_COUNT = 1
  SCALING_POLICY = 'ECONOMY';

ALTER WAREHOUSE my_mcw SET
  MAX_CLUSTER_COUNT = 3
  MIN_CLUSTER_COUNT = 3;

Un multi-cluster warehouse Medium con 3 clusters en modo Maximized corre 2 horas. ¿Cuántos créditos consume? → 3 clusters × 4 créditos/hora × 2 horas = 24 créditos.

AWS: Multi-cluster ≈ Auto Scaling Groups para Redshift Serverless o la feature de "concurrency scaling" de Redshift que agrega clusters temporales.
GCP: Similar al auto-scaling de slots en BigQuery Editions donde puedes configurar baseline y max slots.

Auto-Suspend y Auto-Resume

Auto-Suspend

  • Habilitado por defecto. Suspende el warehouse después de un periodo de inactividad
  • Se recomienda un valor bajo: 5-10 minutos o menos
  • Pero si tus queries tienen gaps de 2-3 min, no pongas auto-suspend en 1-2 min (evita ciclo constante de suspend/resume)
  • Para multi-cluster: auto-suspend aplica a todo el warehouse, no a clusters individuales
  • Para deshabilitar: especifica 0 o NULL en SQL

Auto-Resume

  • Habilitado por defecto. Arranca el warehouse cuando se envía un statement
  • Solo aplica cuando todo el warehouse está suspendido (no clusters individuales)
  • Si necesitas control de costos, desactívalo y resume manualmente

¿Qué pasa con el auto-suspend en un multi-cluster warehouse? → Solo suspende cuando el número mínimo de clusters está corriendo Y no hay actividad por el periodo configurado. No suspende clusters individuales.

Scale Up vs Scale Out

Scale Up = Resize (cambiar tamaño)

  • Mejora performance de queries complejas individuales
  • Se puede hacer en caliente (warehouse corriendo)
  • Los nuevos recursos solo se usan para queries encoladas o nuevas, no para las que ya están corriendo
  • Al reducir tamaño, se pierde cache de los recursos removidos

Scale Out = Agregar clusters (multi-cluster)

  • Mejora concurrencia (más usuarios simultáneos)
  • NO mejora performance de queries individuales lentas
  • Requiere Enterprise Edition

Pregunta clásica: "Tus queries individuales son lentas, ¿qué haces?" → Scale Up (resize a un tamaño mayor). "Tienes muchos usuarios concurrentes y las queries se encolan, ¿qué haces?" → Scale Out (multi-cluster) o crear warehouses separados por equipo.

Resize no es para problemas de concurrencia. Multi-cluster no es para mejorar queries lentas individuales. El examen mezcla estos escenarios para confundirte.

Query Acceleration Service (QAS)

Concepto central

Servicio que offloads porciones del procesamiento de queries "outlier" a recursos compartidos serverless. Reduce el impacto de queries que consumen muchos más recursos que el promedio.

  • Requiere Enterprise Edition
  • Se habilita por warehouse: ENABLE_QUERY_ACCELERATION = TRUE
  • Soporta: SELECT, INSERT, CTAS, COPY INTO <table>
  • Ideal para: ad hoc analytics, large scans con filtros selectivos, workloads con volumen impredecible

Scale Factor

Controla el máximo de recursos serverless que puede usar. Es un multiplicador del tamaño del warehouse:

  • Warehouse Medium (4 cr/hr) con scale factor = 5 → puede gastar hasta 20 créditos/hr adicionales
  • Default: 8
  • Valor 0 = sin límite
  • Para multi-cluster: considera subir el scale factor para cubrir todos los clusters
ALTER WAREHOUSE my_wh SET
  ENABLE_QUERY_ACCELERATION = true
  QUERY_ACCELERATION_MAX_SCALE_FACTOR = 5;

Queries elegibles e inelegibles

  • Elegibles: large scans con agregaciones o filtros selectivos; large scans que insertan muchas filas
  • Inelegibles: pocas particiones para escanear, filtros poco selectivos, GROUP BY con cardinalidad muy alta, funciones no determinísticas (RANDOM, SEQ)

Monitoreo

  • SYSTEM$ESTIMATE_QUERY_ACCELERATION('query_id') — evalúa si una query se beneficiaría
  • SNOWFLAKE.ACCOUNT_USAGE.QUERY_ACCELERATION_ELIGIBLE — vista para encontrar queries/warehouses que se beneficiarían
  • QUERY_ACCELERATION_HISTORY — vista de billing por QAS
  • Query Profile en Snowsight muestra "Partitions scanned by service"

¿Cómo determinas si una query existente se beneficiaría de QAS? → Usa SYSTEM$ESTIMATE_QUERY_ACCELERATION('query_id') o consulta la vista QUERY_ACCELERATION_ELIGIBLE.

AWS: Concepto similar a "burst capacity" o "concurrency scaling" en Redshift, pero QAS es a nivel de query, no de cluster. No hay equivalente directo.
GCP: BigQuery maneja esto internamente con sus slots dinámicos — no necesitas configurarlo manualmente.

QAS y Search Optimization Service son cosas diferentes que pueden trabajar juntas. Search Optimization pruning micro-partitions primero, luego QAS offloads el scanning restante. No los confundas.

Warehouse Cache

Cómo funciona

  • Cada warehouse mantiene un cache de datos de tabla accedidos durante la ejecución de queries
  • El tamaño del cache es proporcional al tamaño del warehouse (más grande = más cache)
  • El cache se pierde al suspender el warehouse
  • Al resumir, las primeras queries serán más lentas hasta que el cache se reconstruya
  • Reducir el tamaño de un warehouse corriendo también elimina cache de los recursos removidos

Trade-off clave

Suspender = ahorra créditos pero pierde cache → queries lentas al volver. Mantener corriendo = gasta créditos pero queries rápidas por cache caliente. Decide según tu workload.

¿Qué pasa con el cache cuando suspendes un warehouse? → Se elimina por completo. Las queries posteriores al resume serán más lentas mientras se reconstruye el cache. Este es el "warehouse cache" (diferente del query result cache y metadata cache).

Monitoreo de Carga

Load Monitoring Chart (Snowsight)

Muestra la carga de queries concurrentes en un warehouse durante hasta 2 semanas. Requiere privilegio MONITOR.

Los estados de queries visibles son:

  • Running: queries ejecutándose activamente
  • Queued (Provisioning): esperando que el warehouse provisione recursos (típico al resumir)
  • Blocked: bloqueadas por transaction locks
  • Queued: esperando porque el warehouse está sobrecargado

Cálculo del Query Load

Se calcula como: tiempo de ejecución de todas las queries en un intervalo ÷ tiempo total del intervalo. Si el load es > 1, significa que hay más de 1 query concurrente en promedio. Si hay queuing significativo, es señal de que necesitas más recursos.

¿Qué acción tomas si el load chart muestra load alto con queries encoladas? → Agregar clusters (multi-cluster) o mover queries a un warehouse separado. Si el load es bajo pero las queries son lentas → Resize up.

Interactive Warehouses & Tables

Nota: Esta feature es GA solo en regiones AWS selectas. El examen COF-C03 podría no testarla directamente, pero familiarízate con los conceptos.

Características clave

  • Optimizado para queries de baja latencia y alta concurrencia
  • Timeout de SELECT por defecto: 5 segundos
  • El warehouse NO se auto-suspende — diseñado para correr continuamente
  • Solo puede consultar interactive tables (no tablas estándar)
  • Interactive tables requieren CLUSTER BY obligatorio
  • No soportan DML (UPDATE, DELETE) — solo INSERT OVERWRITE
  • No tienen fail-safe (pero sí Time Travel)

Default Warehouse para Notebooks

Preview Feature — no será testado hasta que sea GA.

  • Warehouse automático: SYSTEM$STREAMLIT_NOTEBOOK_WH
  • Multi-cluster XS con máximo 10 clusters y timeout de 60 segundos
  • ACCOUNTADMIN tiene OWNERSHIP; PUBLIC tiene USAGE por defecto
  • Best practice: separar workloads Python de Notebooks de queries SQL en warehouses diferentes

Diseño de Warehouses por caso de uso

Caso de usoRecomendación

Billing: reglas que debes saber

  • Mínimo 60 segundos por cada resume
  • Después: per-second
  • Resize a un tamaño mayor → créditos adicionales desde el momento del resize
  • Multi-cluster: créditos = tamaño × clusters activos × tiempo
  • QAS: billing por segundo de recursos serverless usados, separado del warehouse

🪤 Trampas y Confusiones Frecuentes del Examen

"Más grande = más rápido para todo" — FALSO. Queries pequeñas/simples no se benefician de warehouses XL+. Y para data loading, el tamaño tiene impacto limitado.

"Multi-cluster mejora queries lentas" — FALSO. Multi-cluster mejora concurrencia. Para queries lentas individuales necesitas resize (scale up).

"Resize afecta queries en ejecución" — FALSO. Los recursos nuevos solo se usan para queries encoladas y futuras. Las queries actuales terminan con los recursos originales.

"Resource monitors consumen créditos" — FALSO. Los resource monitors rastrean uso de créditos sin costo adicional (ver pregunta de ejemplo en la guía oficial).

"QAS = Search Optimization Service" — Son diferentes. QAS offloads scanning de queries outlier a recursos serverless. Search Optimization pruning micro-partitions con índices. Pueden trabajar juntos: primero SO pruning, luego QAS accelera el resto.

"Economy policy = sin clusters adicionales" — FALSO. Economy SÍ arranca clusters adicionales, pero solo si estima al menos 6 minutos de carga para ese cluster. Puede causar más queuing que Standard.

"Auto-suspend suspende clusters individuales en multi-cluster" — FALSO. Auto-suspend aplica a todo el warehouse, no a clusters individuales. Los clusters individuales se manejan por la scaling policy.

Confundir los 3 tipos de cache: (1) Query Result Cache — 24hrs, a nivel de Cloud Services, no consume warehouse. (2) Metadata Cache — info sobre row count, min/max, etc. automático. (3) Warehouse Cache (local disk/SSD) — datos de tablas recientes, se pierde al suspend. El examen pregunta específicamente cuál se pierde al suspender.

Snowpark-optimized para todo — FALSO. Si tu workload no usa Snowpark (Python/Java/Scala UDFs, ML training), un Snowpark-optimized warehouse no te beneficia y además cuesta más créditos.
