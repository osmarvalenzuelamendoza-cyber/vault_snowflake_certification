Tables, Views & Storage

SnowPro Core COF-C03 · Domains 1.5, 4.2, 4.3, 5.1 · Guía de Estudio Completa

🏆 Top 10 Conceptos Más Importantes

  1. Micro-partitions — Unidades de 50-500 MB (sin comprimir), columnar, automáticas. Son la base de todo el modelo de storage y pruning de Snowflake.
  2. Clustering Keys — Solo para tablas multi-TB. Máximo 3-4 columnas, ordenar de menor a mayor cardinalidad. Genera costos de créditos serverless.
  3. Tabla Permanent vs Transient vs Temporary — Diferencias en Time Travel (0-90 días) y Fail-safe (7 días solo en permanent). Transient y Temporary NO tienen Fail-safe.
  4. External Tables — Read-only, datos en cloud storage externo. Usan VALUE (VARIANT), METADATA$FILENAME. Se pueden particionar. Se refrescan con event notifications.
  5. Hybrid Tables — Row-store, row-level locking, PRIMARY KEY obligatorio y enforced, FOREIGN KEY enforced. Para workloads OLTP + OLAP (Unistore). No soportan clustering keys.
  6. Secure Views — Ocultan definición SQL y evitan inferencia de datos vía optimizaciones. Obligatorias para Secure Data Sharing. Más lentas que views normales.
  7. Materialized Views — Datos pre-computados, mantenimiento automático serverless (costo de créditos). Enterprise Edition+. No soportan JOINs ni UDFs en su definición.
  8. Search Optimization Service — Crea "search access paths". Enterprise Edition+. Para point lookups, VARIANT queries, LIKE/ILIKE, geospatial. Costo serverless independiente del warehouse.
  9. Cloning (Zero-Copy) — Metadatos, no duplica datos. Clustering key se copia pero Automatic Clustering queda suspendido en el clon. Grants NO se copian al container (sí a child objects).
  10. Data Storage Lifecycle — Active → Time Travel → Fail-safe. DROP columna no libera storage inmediatamente. Archivos en stages internos no tienen Time Travel ni Fail-safe.

1. Micro-partitions & Data Clustering

Concepto Clave

Snowflake almacena todos los datos en micro-partitions: unidades contiguas de almacenamiento de 50 a 500 MB de datos sin comprimir. Internamente los datos se guardan en formato columnar y cada columna se comprime de forma independiente con el algoritmo más eficiente.

Snowflake mantiene metadatos por cada micro-partition: rango de valores por columna, número de valores distintos, y propiedades adicionales para optimización. Esto permite query pruning: el motor puede saltar micro-partitions que no contienen datos relevantes para un filtro.

Datos Importantes para el Examen

  • La partición es automática: no requiere DDL ni mantenimiento manual.
  • Las micro-partitions pueden tener rangos que se solapan (overlap) — eso es normal.
  • Columnar storage: solo se escanean las columnas referenciadas por la query.
  • DML (DELETE, UPDATE) usa metadatos de micro-partitions para operar eficientemente.
  • DROP de una columna NO reescribe las micro-partitions; los datos de la columna eliminada permanecen en storage.
  • Snowflake no hace pruning con subqueries en predicados, aunque resuelvan a una constante.

Clustering Depth

Es la métrica que mide el promedio de solapamiento entre micro-partitions para ciertas columnas. Menor depth = mejor clustering. Funciones de diagnóstico:

-- Ver profundidad de clustering
SELECT SYSTEM$CLUSTERING_DEPTH('mi_tabla');

-- Ver información completa de clustering  
SELECT SYSTEM$CLUSTERING_INFORMATION('mi_tabla', '(col1, col2)');
En BigQuery (GCP), el concepto análogo es el partitioning + clustering nativo: BQ particiona por columnas (fecha, integer range) y luego aplica clustering automático sobre hasta 4 columnas. En Snowflake, las micro-partitions son el equivalente al partitioning pero totalmente automáticas y más granulares. En Redshift (AWS), el equivalente son los sort keys + distribución de slices.
P: ¿Cuál es el tamaño de una micro-partition en Snowflake?
R: Entre 50 y 500 MB de datos sin comprimir. El tamaño real es menor porque los datos siempre se almacenan comprimidos.

P: ¿Qué sucede cuando se elimina una columna con ALTER TABLE ... DROP COLUMN?
R: Las micro-partitions NO se reescriben. Los datos de la columna permanecen en storage hasta que las particiones sean reemplazadas por otras operaciones.

2. Clustering Keys & Clustered Tables

Concepto Clave

Un clustering key es un subconjunto de columnas (o expresiones) que se designa explícitamente para co-localizar datos en las mismas micro-partitions. Es para tablas muy grandes (multi-TB) donde el orden natural se ha degradado por DML extensivo.

Reglas Críticas

  • Máximo recomendado: 3-4 columnas por key.
  • Ordenar de menor a mayor cardinalidad en el CLUSTER BY.
  • Para VARCHAR, solo se usan los primeros 5 bytes para clustering.
  • No se soportan tipos: GEOGRAPHY, VARIANT, OBJECT, ARRAY directamente (sí se puede usar una expresión que extraiga un path de VARIANT).
  • Hybrid tables NO soportan clustering keys (usan PRIMARY KEY para ordenamiento).
  • Columnas de alta cardinalidad: usar expresiones como TO_DATE(timestamp_col) o TRUNC(number, -5).
-- Clustering con expresiones para reducir cardinalidad
CREATE TABLE ventas (
  ts TIMESTAMP, region STRING, monto NUMBER
) CLUSTER BY (TO_DATE(ts), region);

-- Clustering sobre paths en VARIANT
CREATE TABLE logs (t TIMESTAMP, v VARIANT) 
  CLUSTER BY (v:"Data":id::number);

-- Cambiar clustering key
ALTER TABLE ventas CLUSTER BY (region, TO_DATE(ts));

-- Eliminar clustering key
ALTER TABLE ventas DROP CLUSTERING KEY;

Cuándo Usar Clustering

  • Tabla con múltiples terabytes de datos.
  • Queries selectivos que leen un porcentaje pequeño de filas.
  • Alto ratio de queries vs. DML (se consulta mucho, se modifica poco).
  • Las mismas columnas aparecen en filtros/joins de la mayoría de queries.

Costo del Reclustering

Reclustering genera nuevas micro-partitions y marca las originales como eliminadas (pero se retienen para Time Travel + Fail-safe). Esto implica costos de créditos serverless + storage temporal adicional.

En BigQuery, el clustering se define con CLUSTER BY col1, col2 en CREATE TABLE, y el re-clustering es automático y gratuito. En Snowflake, el reclustering también es automático pero consume créditos serverless. En Redshift, el equivalente son los SORT KEYS y debes ejecutar VACUUM para reordenar datos.
P: ¿En qué orden deben especificarse las columnas en un CLUSTER BY con múltiples columnas?
R: De menor a mayor cardinalidad. Poner una columna de alta cardinalidad primero reduce la efectividad del clustering en las columnas siguientes.

3. Automatic Clustering

Concepto Clave

Servicio de Snowflake que maneja automáticamente todo el reclustering en background. No requiere warehouse del usuario — usa serverless compute.

Puntos para el Examen

  • Se activa automáticamente al definir un clustering key (excepto en tablas clonadas, donde queda suspendido).
  • Es non-blocking: no interfiere con DML concurrente.
  • Se puede suspender/resumir con ALTER TABLE ... SUSPEND/RESUME RECLUSTER.
  • Cambiar el clustering key de una tabla resume automatic clustering automáticamente (incluso si solo cambias a LINEAR).
  • Resource Monitors NO controlan el crédito del warehouse interno AUTOMATIC_CLUSTERING.
  • Para estimar costos: SYSTEM$ESTIMATE_AUTOMATIC_CLUSTERING_COSTS.
  • Para ver historial de costos: vista AUTOMATIC_CLUSTERING_HISTORY.
-- Suspender automatic clustering
ALTER TABLE t1 SUSPEND RECLUSTER;

-- Resumir automatic clustering  
ALTER TABLE t1 RESUME RECLUSTER;

-- Estimar costos antes de definir clustering
SELECT SYSTEM$ESTIMATE_AUTOMATIC_CLUSTERING_COSTS('mi_tabla', '(col1, col2)');
P: ¿Qué sucede con Automatic Clustering cuando clonas una tabla con clustering key?
R: Se copia el clustering key pero Automatic Clustering queda suspendido en la tabla clonada. Debes resumirlo manualmente.

P: ¿Automatic Clustering usa el warehouse del usuario?
R: No. Usa recursos serverless internos de Snowflake. El cobro es por créditos consumidos, no por warehouse.

5. Temporary & Transient Tables

Temporary Tables

  • Solo existen en la sesión que las creó. No visibles para otros usuarios.
  • Pueden tener el mismo nombre que una tabla permanente — la temporal tiene precedencia en la sesión (¡cuidado con esto!).
  • No requieren privilegio CREATE TABLE en el schema.
  • No se pueden convertir a otro tipo después de creadas.
  • Si mantienes sesiones largas (>24h), las temporales acumulan storage. Recomendación: DROP explícito.

Transient Tables

  • Persisten hasta DROP explícito, visibles para todos con privilegios.
  • Sin Fail-safe = menor costo de storage vs. permanent.
  • Se pueden crear databases y schemas transient — todas las tablas dentro serán transient por definición.
  • Clone de permanent a transient: crea zero-copy clone pero los micro-partitions compartidos no entran a Fail-safe hasta que la transient se elimine.
CREATE TEMPORARY TABLE temp_etl (id NUMBER, data VARIANT);

CREATE TRANSIENT TABLE staging_data (id NUMBER, payload STRING);

-- Database transient (todas las tablas serán transient)
CREATE TRANSIENT DATABASE staging_db;
En BigQuery, las tablas temporales se crean con CREATE TEMP TABLE y expiran al cerrar la sesión (similar). No existe el concepto de "transient" — en BQ usarías table expiration para controlar lifecycle. En Redshift, las tablas temporales son CREATE TEMP TABLE con scope de sesión.
P: ¿Cuál es la diferencia principal entre Transient y Temporary en términos de visibilidad?
R: Temporary = solo visible en la sesión. Transient = visible para cualquier usuario con privilegios, persiste hasta DROP.

P: ¿Qué tipo de Fail-safe tienen las tablas Transient y Temporary?
R: Ninguno. Ambas tienen 0 días de Fail-safe.

6. External Tables

Concepto Clave

Permiten consultar datos en cloud storage externo (S3, GCS, Azure Blob) como si fuera una tabla nativa. Son read-only (no se puede hacer DML).

Columnas Automáticas

  • VALUE — Tipo VARIANT, representa una fila del archivo externo.
  • METADATA$FILENAME — Nombre del archivo staged incluyendo path.
  • METADATA$FILE_ROW_NUMBER — Número de fila en el archivo.

Particionamiento

Se recomienda particionar external tables. Dos modos:

  • Automático: Se definen partition columns como expresiones sobre METADATA$FILENAME. Se actualizan al refrescar metadatos.
  • Manual (USER_SPECIFIED): Se agregan particiones con ALTER EXTERNAL TABLE ... ADD PARTITION. No soporta auto-refresh.

Auto-Refresh de Metadatos

  • AWS: Event notifications de S3 (SQS).
  • GCS: Google Pub/Sub + notification integration.
  • Azure: Event Grid + notification integration.

Performance

  • Más lentas que tablas nativas. Para mejorar: usar materialized views sobre external tables.
  • Para Parquet: considerar Apache Iceberg tables para mejor rendimiento.
  • Archivos recomendados: Parquet 256-512 MB, otros formatos 16-256 MB.
P: ¿Se puede hacer INSERT o UPDATE en una external table?
R: No. Las external tables son estrictamente read-only.

P: ¿Cómo se mejora el rendimiento de queries sobre external tables?
R: Creando materialized views sobre la external table (requiere Enterprise Edition).

7. Hybrid Tables

Concepto Clave

Tipo de tabla optimizado para baja latencia y alto throughput con operaciones OLTP. Usa un row store como storage primario con copia asíncrona a columnar en object storage para analytics.

Características Únicas

FeatureHybridStandard
Layout primarioRow-orientedColumnar (micro-partitions)
LockingRow-levelPartition/table
PRIMARY KEYObligatorio, enforcedOpcional, NOT enforced
FOREIGN KEYEnforcedNOT enforced
UNIQUEEnforcedNOT enforced
IndexesSoportados (sync)Solo Search Optimization (async)
Clustering KeysNO soportadoSoportado

Limitaciones

  • No se pueden crear como TEMPORARY o TRANSIENT.
  • No se pueden crear en schemas/databases transient.
  • Mayor footprint de storage (row store comprime menos que columnar).
  • Disponible solo en AWS y Azure commercial regions.
Las hybrid tables son el equivalente de Snowflake a un Cloud SQL / Cloud Spanner (GCP) o Aurora / DynamoDB (AWS) pero integrado en la misma plataforma. El concepto de "Unistore" combina OLTP + OLAP sin federación — algo que en GCP/AWS requeriría CDC entre servicios separados.
P: ¿Qué constraint es OBLIGATORIO en una hybrid table?
R: PRIMARY KEY. Es la única tabla donde es requerido.

P: ¿Se puede definir un clustering key en una hybrid table?
R: No. Los datos siempre se ordenan por el PRIMARY KEY.

8. Views (Standard / Non-Materialized)

Concepto Clave

Una view es una definición con nombre de una query. No almacena datos — se ejecuta cada vez que se referencia. Usa el schema del momento en que fue creada (no el schema activo).

Puntos para el Examen

  • Se pueden usar en JOINs, subqueries, casi en cualquier lugar donde se use una tabla.
  • Si no especificas schema, Snowflake asume que la tabla está en el mismo schema que la view.
  • No almacenan datos = no hay costo de storage ni mantenimiento.
  • La definición (SQL) es visible por defecto para usuarios con acceso (a diferencia de secure views).

9. Secure Views

Concepto Clave

Una secure view oculta la definición SQL y desactiva ciertas optimizaciones internas para prevenir inferencia de datos.

Por Qué Existen

  • En una view normal, optimizaciones internas pueden exponer datos ocultos (ej: inferencia por division-by-zero).
  • La definición SQL de una view normal es visible en SHOW VIEWS, GET_DDL, etc.
  • En una secure view, la definición solo es visible para el OWNER del view.

Cuándo Usar

  • Para data privacy y limitar acceso a datos sensibles.
  • Obligatorio para Secure Data Sharing (compartir datos entre cuentas).
  • NO usar solo por conveniencia de queries — son más lentas que views normales.
-- Crear secure view
CREATE SECURE VIEW sensitive_data_vw AS
  SELECT id, masked_email FROM customers;

-- Convertir view existente a secure
ALTER VIEW my_view SET SECURE;

-- Quitar secure
ALTER VIEW my_view UNSET SECURE;
P: ¿Por qué una secure view podría ser más lenta que una view normal?
R: Porque desactiva ciertas optimizaciones internas del query optimizer para prevenir inferencia de datos.

P: ¿Quién puede ver la definición SQL de una secure view?
R: Solo el rol que es OWNER de la view.

10. Materialized Views

Concepto Clave

Datos pre-computados y almacenados derivados de una query. Se actualizan automáticamente cuando cambian los datos base. Requieren Enterprise Edition.

Cuándo Usar

  • Query results con pocas filas/columnas relativo a la tabla base.
  • Procesamiento costoso: agregaciones, semi-structured data analysis.
  • La tabla base cambia poco y la view se consulta frecuentemente.
  • Sobre external tables para mejorar rendimiento.

Ventajas

  • Mantenimiento automático por background service (serverless).
  • Datos siempre actuales: si no está al día, combina cache + base table.
  • Soportan clustering (a diferencia de views normales).

Limitaciones

  • No soportan JOINs en la definición.
  • Solo pueden consultar una tabla.
  • No soportan UDFs, HAVING, ORDER BY, LIMIT, window functions en la definición.
  • Costo de storage + créditos serverless para mantenimiento.

Comparativa Rápida

En BigQuery, las materialized views existen nativamente y se actualizan automáticamente (sin costo extra de mantenimiento, solo storage). En Redshift, las materialized views requieren REFRESH MATERIALIZED VIEW manual o Auto-Refresh. Snowflake es más similar a BigQuery en que el refresh es automático, pero cobra créditos serverless.
P: ¿Se puede crear una materialized view con JOIN?
R: No. Las materialized views solo pueden hacer referencia a una tabla base. Si necesitas JOINs, usa Dynamic Tables.

P: ¿Qué edición de Snowflake requiere materialized views?
R: Enterprise Edition o superior.

11. Search Optimization Service

Concepto Clave

Crea y mantiene un search access path — una estructura persistente que trackea qué valores existen en cada micro-partition. Requiere Enterprise Edition.

Tipos de Queries que Beneficia

  • Point lookups selectivos (WHERE col = 'value').
  • Búsquedas con LIKE, ILIKE, RLIKE (substrings y regex).
  • Queries sobre elementos en VARIANT, OBJECT, ARRAY.
  • Búsquedas con SEARCH() y SEARCH_IP().
  • Queries geospatiales con GEOGRAPHY.

Activación

-- Habilitar para toda la tabla
ALTER TABLE mi_tabla ADD SEARCH OPTIMIZATION;

-- Habilitar para columnas específicas
ALTER TABLE mi_tabla ADD SEARCH OPTIMIZATION 
  ON EQUALITY(col1), SUBSTRING(col2);

-- Deshabilitar
ALTER TABLE mi_tabla DROP SEARCH OPTIMIZATION;

Costos

  • Usa serverless compute para construcción y mantenimiento.
  • Consume storage adicional para el search access path.
  • Es independiente del warehouse del usuario.

Search Optimization vs Clustering Keys

AspectoSearch OptimizationClustering Keys
Mejor paraPoint lookups, wildcardsRange scans, queries de rango
EstructuraSearch access pathReordenamiento de micro-partitions
Se pueden combinarSí — son complementarios
P: ¿Search Optimization y Clustering Keys son mutuamente excluyentes?
R: No. Son complementarios. Clustering es mejor para range scans; Search Optimization es mejor para point lookups y búsquedas exactas.

12. Cloning (Zero-Copy Clone)

Concepto Clave

Snowflake crea un clon a nivel de metadatos (zero-copy) que comparte las micro-partitions del objeto original. No duplica datos hasta que se modifican (copy-on-write).

Reglas para el Examen

  • Se pueden clonar: databases, schemas, tables (no external tables), stages, file formats, sequences, streams, tasks.
  • Grants en child objects SÍ se copian al clonar database/schema; grants en el container NO se copian.
  • Soporte COPY GRANTS explícito en CREATE TABLE ... CLONE.
  • Clustering key se copia, pero Automatic Clustering queda SUSPENDIDO.
  • Si la tabla tiene secuencias default y se clona el schema/database, el clon referencia la secuencia clonada; si se clona solo la tabla, referencia la secuencia original.
  • Foreign keys: si se clona el schema/database, apuntan a la tabla clonada del PK; si se clona solo la tabla, apuntan a la tabla original.

Cloning y Tipos de Tablas

SourceTarget posible
PermanentPermanent, Transient, Temporary
TransientTransient, Temporary
TemporaryTemporary, Transient
En BigQuery, el equivalente es CREATE TABLE ... CLONE que también es zero-copy. En AWS (Redshift), no hay cloning nativo equivalente — necesitarías UNLOAD/COPY o snapshots. La ventaja de Snowflake es que el clone es instantáneo y aplica a toda la jerarquía (DB > Schema > Tables).
P: Al clonar una tabla con clustering key, ¿qué pasa con Automatic Clustering?
R: Se copia el clustering key pero Automatic Clustering queda suspendido. Debes ejecutar ALTER TABLE ... RESUME RECLUSTER para activarlo.

13. Data Storage Considerations

Lifecycle de Datos (CDP)

Los datos pasan por tres estados: Active → Time Travel → Fail-safe. Los tres generan costos de storage.

Puntos para el Examen

  • Time Travel: 0-1 día (Standard), 0-90 días (Enterprise+) para tablas permanentes. Transient/Temporary: máximo 1 día.
  • Fail-safe: 7 días fijos (no configurable). Solo tablas permanentes. Snowflake accede a Fail-safe; el usuario no puede.
  • DROP columna: datos permanecen en storage (no se reescriben micro-partitions).
  • Archivos en stages internos: no tienen Time Travel ni Fail-safe, pero sí generan costo de storage estándar.
  • Usar TABLE_STORAGE_METRICS (Information Schema) para ver desglose de ACTIVE_BYTES, TIME_TRAVEL_BYTES, FAILSAFE_BYTES.
  • Reclustering genera turnover de micro-partitions → incrementa TIME_TRAVEL_BYTES y FAILSAFE_BYTES temporalmente.
P: ¿El periodo de Fail-safe es configurable?
R: No. Siempre es 7 días para tablas permanentes, 0 para transient y temporary.

P: ¿Los archivos en internal stages tienen Fail-safe?
R: No. Solo generan costos de storage estándar. Se recomienda eliminarlos después de cargar datos.

14. Table Design Considerations

Mejores Prácticas

  • Elegir tipos de datos apropiados: usar NUMBER para enteros en lugar de VARCHAR. Los tipos más eficientes mejoran la compresión.
  • Definir clustering keys solo en tablas de varios TB — en tablas pequeñas el costo supera el beneficio.
  • Usar TRANSIENT para datos que se pueden reconstruir (staging, ETL intermedio).
  • Usar TEMPORARY para datos de sesión que no necesitan persistir.
  • Considerar Dynamic Tables en lugar de pipelines streams+tasks cuando sea posible (declarativo vs imperativo).
  • Para datos semi-structured: VARIANT almacena bien JSON, pero para queries frecuentes sobre paths específicos, considerar flatten + columnas tipadas.

🪤 Trampas y Confusiones Frecuentes del Examen

Temporary vs Transient: Ambas no tienen Fail-safe. La diferencia es la persistencia: temporary = solo sesión; transient = hasta DROP explícito. No confundir.
Materialized Views NO soportan JOINs: Si la pregunta habla de pre-computar resultados de un JOIN, la respuesta NO es materialized view — es Dynamic Table.
Clustering Key: orden de columnas importa: Siempre menor a mayor cardinalidad. Muchos responden instintivamente poner la columna más selectiva primero (como en un índice B-tree) — en Snowflake es al revés.
External Tables son READ-ONLY: No puedes hacer INSERT, UPDATE, DELETE, MERGE. Tampoco tienen Time Travel ni Fail-safe.
Hybrid Tables: PK obligatorio y enforced: Es el ÚNICO tipo de tabla donde PRIMARY KEY es requerido Y enforced. En tablas estándar, los constraints son opcionales y NOT enforced (excepto NOT NULL).
Secure View ≠ mejor performance: Las secure views son MÁS LENTAS porque desactivan optimizaciones. No usar por conveniencia, solo por seguridad/privacidad.
Cloning: Automatic Clustering queda suspendido: Muchos asumen que al clonar una tabla con clustering, todo sigue funcionando igual. No. Debes resumir manualmente.
Search Optimization ≠ Clustering: Son complementarios. Search Optimization es para point lookups (equality, LIKE); Clustering es para range scans. Ambos usan serverless compute y tienen costos separados.
DROP COLUMN no libera storage: Los datos de la columna eliminada permanecen en las micro-partitions existentes hasta que sean reemplazadas por DML posterior.
Resource Monitors NO controlan costos serverless: Automatic Clustering, Search Optimization, y Materialized View maintenance usan serverless compute que NO se puede limitar con Resource Monitors.
